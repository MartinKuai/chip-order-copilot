# 电子元器件订单执行风控 Copilot - 产品与架构规划

本项目旨在构建一个贴近真实 B2B 业务场景的 AI Demo，帮助转行 AI 产品/售前岗位的候选人在面试中展示“将行业痛点抽象为可落地 AI 应用”的能力。

## 一、 产品定位

- **产品名称**: 电子元器件订单执行风控 Copilot
- **核心价值定位**: 面向芯片代理商 B2B 订单执行场景的 AI 协同助手。它通过 LLM 将销售、客户、助理之间非结构化的碎片沟通记录，转化为结构化的订单交接卡片，并结合结构化台账（订单、库存、报价）与预设业务规则进行风控校验。
- **目标达成**: 帮助销售助理在执行发货前，自动识别信息缺口（如：缺失标签、无正式PO）和业务风险（如：报价过期、库存被占），从而减少错发漏发、避免合规争议和内部扯皮。

## 二、 用户画像

1. **核心用户（使用者）：销售助理 (Sales Assistant)**
   - **痛点**: 经常在信息黑盒中接手任务，对历史沟通不知情；容易遗漏后置的特殊要求；无权限确认价格/库存状态；出了问题容易成为“背锅侠”。
   - **诉求**: 明确当前订单能不能发、缺什么条件、风险在哪里。
2. **上游用户：销售 (Sales)**
   - **痛点**: 在外应酬或开会，没时间填复杂的审批单或交接表。
   - **诉求**: 复制粘贴几句聊天记录就能把活交接出去。
3. **管理用户：销售经理 / 商务负责人**
   - **诉求**: 订单流转透明化，系统级规避价格倒挂和违规发货风险。
4. **下游用户：库房 / 物流**
   - **诉求**: 只看明确的指令（贴什么签、怎么包装、发哪里）。

## 三、 核心业务流程

1. **信息输入**: 销售或助理将企业微信/微信中的多方群聊记录复制粘贴至 Copilot。
2. **上下文挂载**: 助理选择或上传对应的结构化业务数据（订单台账、库存台账、报价记录）。
3. **AI 抽取**: LLM 从杂乱的对话中提取关键业务实体（型号、数量、特殊包装/标签要求、交期期望等）。
4. **规则融合校验**: 系统将 AI 提取的内容与结构化台账进行 Join，通过本地业务规则引擎进行逻辑判断（如：提取到价格为 $1.2，校验台账发现报价已过期）。
5. **卡片生成与协同**: Copilot 输出带红绿灯状态的风险报告、发货 Checklist，并生成一份规范的 Markdown/文本格式交接卡片。
6. **分发**: 助理一键复制卡片，发送至内部执行群，各环节按卡片指令行事。

## 四、 Demo 主线设计

为保证演示的“半真实”和“动态可变”特性，Demo 的交互主线设计如下：

1. **场景预设切换**: 提供 3-4 个典型业务场景按钮（如“常规缺货场景”、“特殊标签定制场景”、“报价过期强行发货场景”）。点击后自动填充对应的沟通记录和背景数据表（免去面试时当场手敲的尴尬，但也支持手动修改以证明不是硬编码）。
2. **一键分析**: 用户点击“开始分析”。
3. **过程展示**: 页面显示 AI 思考中（提取字段... 匹配库存... 校验规则...），体现工作流流转感。
4. **结果呈现**: 
   - 顶部展示：核心订单摘要及综合风险评级（高危/警告/安全）。
   - 左侧展示：沟通记录原文 + 高亮提取出的关键信息。
   - 右侧展示：业务规则校验结果（Checklist）及风险解释。
5. **行动输出**: 底部或侧边栏生成“标准化交接卡片”，提供“一键复制”功能。

## 五、 MVP 功能边界

为保证 Demo 在短期内高质高效交付，同时避免陷入庞大的 ERP 泥潭，严格划定边界：

> [!IMPORTANT] 
> **必须包含 (In Scope)**
> - 纯文本聊天记录输入框。
> - 内置多套高仿真的 JSON/CSV Mock 数据源（台账/库存/报价），支持下拉切换，并预留本地文件上传口。
> - 基于 LLM 的非结构化字段抽取能力。
> - 纯前端/Serverless 层面的规则引擎判断逻辑。
> - 风险可视化面板（红绿灯、Checklist）。
> - 结构化交接卡片生成及剪贴板复制功能。
> - 完善的错误兜底逻辑（Mock 模式切换）。

> [!WARNING]
> **绝对不碰 (Out of Scope)**
> - 任何真实的 ERP/CRM 系统 API 对接。
> - 用户登录注册、RBAC 权限管理。
> - 数据库持久化存储（刷新页面可丢失数据，纯展示用）。
> - 真实的审批流流转动作（仅生成建议）。
> - 复杂的自动分货、自动锁库算法（仅基于当前状态做风险提示）。

## 六、 数据结构设计

建议采用 JSON 格式在前端或 API 层进行流转，分为 5 个核心领域：

### 1. Context Data (输入层)
```json
{
  "chat_record": "客户：A133要5000颗，最好明天发。销售：库存有，让助理建单。客户：外箱要贴客户自己的标签，不能混批。..."
}
```

### 2. Ledger Data (台账层 - 订单/客户)
```json
{
  "order_id": "PO-202405-001",
  "customer_id": "CUST_088",
  "customer_name": "某知名终端制造厂",
  "sales_rep": "张三",
  "has_official_po": false,          // 缺失正式PO
  "address_confirmed": true,
  "label_template_available": false, // 缺失标签模板
  "allow_mix_batch": false
}
```

### 3. Inventory & Price Data (库存与报价层)
```json
{
  "inventory": {
    "part_num": "A133",
    "total_avail": 3000,             // 数量不足
    "batches": [{"batch_no": "2345", "qty": 3000}]
  },
  "price": {
    "part_num": "A133",
    "last_quoted": "2024-03-01",
    "valid_days": 30                 // 已过期
  }
}
```

### 4. Extracted Data (LLM 抽取层)
```json
{
  "extracted_part": "A133",
  "requested_qty": 5000,
  "urgency": "High",
  "special_reqs": ["贴客户标签", "不可混批"],
  "missing_info": ["正式PO", "标签模板", "收货人联系方式"]
}
```

## 七、 页面结构设计

采用现代化的高级控制台 (Dashboard) 风格，注重数据密集型应用的 UI 美感。遵循原生 CSS/CSS Modules 方案，构建灵活且极具质感的交互。

### 1. 首页 / 场景引导页 (Hero Section)
- **视觉**: 简洁的深色模式或充满科技感的毛玻璃效果 (Glassmorphism)。使用现代无衬线字体（如 Inter）。
- **内容**: 痛点三连问（“销售不在？价格过期？特殊标签没交代？”），一句话产品定位，“进入工作台” CTA 按钮。

### 2. 分析工作台 (Main Workspace) - 左右分栏布局
- **左侧 (Input & Context)**:
  - **顶部**: 预设场景一键体验按钮组。
  - **中间**: 沟通记录文本域（高亮输入区）。
  - **底部**: 数据源状态面板（通过折叠面板展示当前挂载的订单/库存 Mock 数据状态）。
  - **动作**: 醒目的“执行风控分析”主按钮（带有微动画效果）。
- **右侧 (Output & Analysis)**: (分析前展示骨架屏)
  - **区块 A (订单摘要)**: 抽取出的型号、数量、交期要求卡片。
  - **区块 B (风控大屏)**:
    - 风险等级提示框（基于预警级别展示红/黄/绿）。
    - 风险详情列表（如：🔴 [库存风险] 需求 5000 颗，当前可用仅 3000 颗；🔴 [合规风险] 要求贴签但系统无模板）。
  - **区块 C (Checklist)**: 发货前必做事项打勾列表。

### 3. 结果交付区 (Action Panel)
- 悬浮在右侧底部或单独的模态框，展示最终的 **业务交接卡片**。
- 提供“复制到企业微信”按钮，格式排版友好。

## 八、 AI Prompt 设计

Prompt 需要采用系统化的角色扮演和约束指令：

```text
# Role
你是一个资深的电子元器件代理商订单风控专家和高级销售助理。你的任务是从散乱的聊天记录中提取订单关键信息，并严格输出为结构化 JSON。

# Background
在半导体分销行业，订单常涉及型号(MPN)、批次(DateCode/Lot)、环保包装(Tape&Reel/Tray)、客户专属标签、分批发货等复杂属性。

# Task
请阅读以下聊天记录：
<chat_record>
{{INPUT_CHAT}}
</chat_record>

# Extraction Rules
1. 提取所有涉及的型号和数量。
2. 识别是否有特殊包装或标签要求（如：不拆包、特定原厂标签、不可混批）。
3. 识别交期紧急程度。
4. 识别当前缺失的重要信息（在正常订单流中必须要有但聊天里没提的，如收货人、PO号、是否含税等）。

# Output Format
严格按照以下 JSON Schema 输出，不要包含任何 Markdown 标记或多余解释：
{
  "entities": [
    {
      "part_number": "string",
      "quantity": "number",
      "special_requirements": ["string"]
    }
  ],
  "urgency_level": "NORMAL|HIGH",
  "identified_missing_info": ["string"]
}
```

## 九、 规则引擎设计

提取完结构化数据后，不完全依赖 LLM 判断业务逻辑（防止幻觉），而是交由本地纯函数构成的规则引擎进行二次判断：

```javascript
// 规则示例逻辑 (纯函数，易于单元测试)
function runRiskEngine(extractedData, ledgerData) {
  let risks = [];
  
  // 规则 1：标签风控
  if (extractedData.special_reqs.includes('贴标签') && !ledgerData.label_template_available) {
    risks.push({ level: 'HIGH', msg: '客户要求贴专属标签，但系统缺失标签模板文件，贸然发货可能导致拒收。', owner: 'Sales' });
  }
  
  // 规则 2：库存风控
  if (extractedData.quantity > ledgerData.inventory.total_avail) {
    risks.push({ level: 'HIGH', msg: `现货库存不足！需求 ${extractedData.quantity}，可用 ${ledgerData.inventory.total_avail}。需确认后续到货期。`, owner: 'Purchasing/Sales' });
  }

  // 规则 3：PO风控
  if (!ledgerData.has_official_po) {
    risks.push({ level: 'MEDIUM', msg: '未收到正式 PO 文件，当前仅为沟通意向，严禁发货。', owner: 'Sales Assistant' });
  }
  
  return risks;
}
```

## 十、 API 和前后端架构

为追求极速加载和零维护成本，采用全 Serverless 架构：

- **Frontend**: `React` (使用 Vite 构建) + 原生 CSS / CSS Modules。采用深色高级质感设计，杜绝简陋感。
- **Backend API Proxy**: `Cloudflare Pages Functions` 提供无服务器 API 路由（例如 `/api/analyze`）。
- **LLM Integration**: 在 CF Function 中通过 Fetch 调用小米 MiMo API 或其他模型 API（避免前端泄露 API Key）。
- **State Management**: React 原生的 `useState` / `useContext` 足矣。

## 十一、 Cloudflare 部署建议

1. **环境变量**: 将 `LLM_API_KEY` 和 `LLM_BASE_URL` 配置在 Cloudflare Pages 的 Environment variables 中，**严禁**直接暴露在前端代码。
2. **超时限制**: CF Pages Functions 免费版有 10秒 运行时间限制。
   - **应对策略**: 优先选用首字节响应快的模型。为防止演示翻车，前端必须做好 Timeout Catch 机制。
3. **路由重写**: 确保设置正确的 fallback 规则，以便处理单页应用的路由及 API 请求。

## 十二、 风险与盲点

> [!CAUTION]
> 1. **LLM 提取幻觉风险**: 对于极其口语化的缩写（如“那颗料”、“跟上次一样”），LLM 可能提取失败。**对策**: 页面上允许用户直观地编辑或修正 AI 提取出的结构化标签。
> 2. **演示网络风险**: 面试现场可能由于网络或 API 抖动导致演示失败。**对策**: 提供一个明显的 "演示兜底模式 (Mock API)" 开关，开启后不请求真实 API，直接毫秒级返回预置的理想 JSON 结果以保障演示流畅。
> 3. **UI 视觉表现力不足**: 缺乏打磨的 B端 Demo 极容易显得像简陋的毕设。**对策**: 高度重视前端视觉，严格选用高级配色方案，运用 CSS 渐变、微动画和优质排版，打造“贵价”的 AI Copilot 质感。

## 十三、 验收清单 (Acceptance Criteria)

- [ ] **数据变异性**: 输入不同的沟通记录，AI 必须准确提取出对应的不同型号和数量。
- [ ] **规则触发 (标签)**: 模拟“缺少标签模板”时，风险大屏必须亮红灯提示“标签风险”。
- [ ] **规则触发 (报价)**: 挂载过期报价数据时，必须阻断发货并提示“不能直接报价，需重新核价”。
- [ ] **规则触发 (PO)**: 订单状态缺失 PO 时，交接状态必须为“等待 PO 确认/不可发货”。
- [ ] **规则触发 (库存)**: 当订单需求数量超出可用库存时，必须触发“缺货高危风险”警报。
- [ ] **规则触发 (批次)**: 当输入要求“不可混批”，但查验到库存由散乱小批次组成时，必须拦截。
- [ ] **兜底健壮性**: 故意配置错误 API Key 后，点击分析页面不能 500 崩溃，需友好提示并无缝降级至 Mock 展示。
- [ ] **成果交付物**: 页面生成的“订单交接卡片”能够完美复制并粘贴到企微/飞书，且排版清晰，核心要素（客户、销售、状态、型号、风险点、行动建议）一应俱全。
